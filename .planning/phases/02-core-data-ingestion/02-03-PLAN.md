<objective>
Implement the Meta Ads API connector and normalizer that pulls campaign hierarchy and daily metrics, stores raw responses in raw_api_pulls, and normalizes them into campaign_metrics using upsert semantics.

Purpose: Meta Ads is one of the three primary data sources. This connector enables INTG-01 — users connecting their Meta Ads account and seeing campaign/ad set/ad data appear in the system. The two-stage raw-to-normalized pipeline (per RESEARCH.md Pattern 1) ensures raw data can be re-normalized if attribution rules change.

Output: A Meta connector that fetches campaigns and metrics, and a normalizer that writes to both raw_api_pulls and campaign_metrics.
</objective>

**packages/ingestion/src/connectors/meta.ts:**

Implement `MetaConnector` implementing the PlatformConnector interface from connector-base.ts. Use `facebook-nodejs-business-sdk` v23.

**fetchCampaigns(config):**
1. Initialize `FacebookAdsApi` with the decrypted access token from config.credentials
2. Create `AdAccount` instance using `act_${config.credentials.metadata.adAccountId}`
3. Call `account.getCampaigns([Campaign.Fields.id, Campaign.Fields.name, Campaign.Fields.status], { limit: 500 })`
4. For each campaign, also fetch ad sets via `campaign.getAdSets(...)` and ads via `adSet.getAds(...)`
5. Return array of RawCampaignData with campaign hierarchy
6. Wrap all API calls in `p-retry` with 5 retries, 30s base timeout, randomize jitter (per RESEARCH.md Pattern 5). Use `AbortError` for non-retryable errors (error code 100 = invalid parameter). Retry on error codes 17 and 613 (rate limits).

**fetchMetrics(config, dateRange):**
1. For date ranges > 7 days, use Meta async reporting (`async: true` parameter) — per RESEARCH.md Pitfall 7
2. Call `account.getInsightsAsync()` with fields: `['spend', 'impressions', 'clicks', 'cpc', 'cpm', 'ctr']`, params: `{ time_range: { since: dateRange.start, until: dateRange.end }, level: 'campaign', time_increment: 1 }` (daily granularity)
3. For async jobs: poll `asyncJob.get(['async_percent_completion', 'async_status'])` every 60 seconds, max 60 attempts (1 hour timeout)
4. When complete, retrieve results via `asyncJob.getResult()`
5. For date ranges <= 7 days, use synchronous `account.getInsights()` instead
6. Apply `p-limit(3)` for concurrency control — no more than 3 concurrent API calls per account
7. Return array of RawMetricData with daily rows

**refreshTokenIfNeeded(config):**
1. Meta long-lived tokens last ~60 days. Check if `tokenExpiresAt` is within 7 days of expiry
2. If so, exchange for a new long-lived token via `https://graph.facebook.com/v23.0/oauth/access_token?grant_type=fb_exchange_token&...`
3. Return updated credentials (or unchanged if no refresh needed)

**CRITICAL considerations from RESEARCH.md:**
- Only request aggregate totals (spend, impressions, clicks) for historical backfill — NOT unique-count fields or breakdowns (Pitfall 2: 13-month limit for breakdown metrics)
- Store attribution window parameter alongside every API call (Pitfall 1: attribution window data incompatibility)
- Use p-retry with exponential backoff + jitter for all API calls (Pattern 5)

**packages/ingestion/src/connectors/index.ts:**

```typescript

```
Import and register MetaConnector. Google Ads and Shopify will be registered in Plans 04 and 05.

**packages/ingestion/src/normalizers/meta.ts:**

Implement the two-stage pipeline per RESEARCH.md Pattern 1:

**Stage 1 — `storeRawPull(params: { tenantId, source: 'meta', apiVersion: 'v23.0', attributionWindow, apiParams, payload }): Promise<string>`:**
1. Insert the raw API response into `rawApiPulls` using Drizzle, with `normalized: false`
2. Use `withTenant` for RLS context (per Phase 1 Pitfall 6 — RESEARCH.md)
3. Return the raw_api_pulls record ID

**Stage 2 — `normalizeMetaInsights(params: { tenantId, rawPullId, payload }): Promise<number>`:**
1. Parse the Meta insights payload — extract daily campaign-level rows
2. For each row, map Meta fields to the NormalizedMetric shape:
   - `date` from `date_start`
   - `campaignId` — look up the internal UUID from the campaigns table by matching `externalId` to Meta's campaign ID and `source = 'meta'`
   - `source` = `'meta'`
   - `spendUsd` from `spend` (Meta reports in account currency — store as-is for v1, currency normalization is a v2 concern)
   - `impressions` from `impressions`
   - `clicks` from `clicks`
   - `ctr` from `ctr` (Meta provides this as a percentage string — parse to decimal)
   - `cpm` from `cpm`
   - `directRevenue`, `directConversions`, `directRoas` = NULL (Meta is spend source, not revenue source — revenue comes from Shopify in Plan 05)
3. Upsert into `campaignMetrics` using `onConflictDoUpdate` with target `[tenantId, campaignId, date, source]` — all four columns as per RESEARCH.md Pitfall 8
4. Update the `rawApiPulls` record: set `normalized = true`, `normalizedAt = now()`, `schemaVersion = '1.0'`
5. Return count of metrics rows upserted

**Export `processMetaSync(params: { tenantId, integrationId, dateRange })` as the top-level orchestrator:**
1. Get connector config (decrypt tokens from integrations table)
2. If campaign hierarchy not recently synced, call `fetchCampaigns()` and upsert into campaigns/adSets/ads tables
3. Call `fetchMetrics()` for the date range
4. Call `storeRawPull()` for the raw response
5. Call `normalizeMetaInsights()` to transform and upsert
6. Update `ingestionCoverage` for each date in the range with status and record count
7. Return summary: { recordsIngested, datesProcessed }

Use `withTenant(tenantId, ...)` for all database operations — the worker MUST set tenant context (RESEARCH.md Pitfall 6).

Validate incoming API payloads with a Zod schema before writing to raw_api_pulls — catches API schema changes before they corrupt data (per RESEARCH.md standard stack: zod for runtime validation).
