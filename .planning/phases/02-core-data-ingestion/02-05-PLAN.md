Implement the Shopify connector and normalizer that pulls order/revenue data via the GraphQL Admin API, uses bulk operations for historical backfill, and writes revenue as direct attribution (directRevenue, directConversions) to campaign_metrics.

Purpose: Shopify provides the ground-truth revenue data that becomes the outcome variable for incrementality analysis. While Meta and Google Ads provide spend (the input), Shopify provides revenue (the output). This is the most important data source for the core value proposition — campaign-level incremental lift analysis.

Output: A Shopify connector handling both incremental syncs (recent orders) and bulk backfill (historical JSONL download), plus a normalizer that maps orders to the direct attribution columns.
</objective>

**packages/ingestion/src/connectors/shopify.ts:**

Implement `ShopifyConnector` implementing PlatformConnector. Use `@shopify/shopify-api` with `@shopify/shopify-api/adapters/node`.

**Initialization:**
Configure `shopifyApi()` with:
- `apiKey` from `SHOPIFY_API_KEY`
- `apiSecretKey` from `SHOPIFY_API_SECRET`
- `scopes` = `['read_orders', 'read_all_orders']` — CRITICAL: `read_all_orders` required for historical backfill (Pitfall 3 from RESEARCH.md)
- `hostName` from `NEXT_PUBLIC_APP_URL`
- `apiVersion` = `LATEST_API_VERSION`
- `isEmbeddedApp` = `false`

**fetchCampaigns(config):**
- Shopify does not have a campaign concept — this is a no-op that returns an empty array. Campaign associations are derived from UTM parameters on orders, which is a Phase 3/4 concern. For Phase 2, Shopify data flows into campaign_metrics as daily revenue totals per store (not per campaign).

**fetchMetrics(config, dateRange) — incremental sync path:**
For daily/incremental syncs (last 1-7 days), use standard GraphQL queries:
```graphql
{
  orders(first: 250, query: "created_at:>='${start}' created_at:<='${end}'") {
    edges {
      node {
        id
        processedAt
        totalPriceSet { shopMoney { amount currencyCode } }
        subtotalPriceSet { shopMoney { amount currencyCode } }
        totalDiscountsSet { shopMoney { amount currencyCode } }
      }
    }
    pageInfo { hasNextPage endCursor }
  }
}
```
Paginate through all pages using cursor-based pagination.
Apply `p-retry` with 5 retries for transient failures.

**fetchMetricsBulk(config, dateRange) — backfill path:**
For historical backfill (date range > 30 days), use Shopify Bulk Operations (per RESEARCH.md Pattern 6):

1. Submit `bulkOperationRunQuery` mutation with orders query filtered by date range
2. Poll `bulkOperation(id:)` every 30 seconds for small stores, 60 seconds for large (>10k orders)
3. When `status === 'COMPLETED'`, download the JSONL file from the `url` field
4. Stream-parse the JSONL file line-by-line to avoid OOM on large datasets — use Node.js `readline` + `createReadStream`. Do NOT `JSON.parse()` the entire file.
5. Return parsed order data

**refreshTokenIfNeeded(config):**
Handle Shopify expiring offline tokens (Pitfall 4 from RESEARCH.md):
1. Check if access token has expired (1-hour lifetime from Dec 2025)
2. If expired, use the refresh token to obtain a new access token via Shopify's token refresh endpoint
3. Update the integrations table with the new encrypted access token and new expiry
4. If refresh token fails (>90 days old), return an error status indicating re-authorization is required — this triggers the 'expired' status on the integration

**Update connectors/index.ts:**
Register `ShopifyConnector` for platform `'shopify'`.

**packages/ingestion/src/normalizers/shopify.ts:**

Follow the two-stage pipeline pattern:

**Stage 1 — `storeRawPull(params)`:** Same pattern as Meta/Google — insert raw order data into rawApiPulls with `source: 'shopify'`, `normalized: false`. Use withTenant for RLS.

**Stage 2 — `normalizeShopifyOrders(params: { tenantId, rawPullId, payload }): Promise<number>`:**
Shopify provides REVENUE data (the outcome variable), not SPEND. This maps to the DIRECT attribution columns in campaign_metrics:

1. Aggregate orders by date — group all orders by their `processedAt` date
2. For each date, calculate:
   - `directRevenue` = sum of `totalPriceSet.shopMoney.amount` for all orders on that date
   - `directConversions` = count of orders on that date
   - `directRoas` = NULL (ROAS requires spend data from Meta/Google, which is computed at query time or by Phase 3)
3. Map to campaign_metrics insert:
   - `date` = the order date
   - `campaignId` — for Phase 2, use a synthetic "all-shopify" campaign per tenant (create in campaigns table with `externalId: 'shopify-revenue'`, `source: 'shopify'`). Per-campaign revenue attribution requires UTM tracking which is a Phase 3/4 concern.
   - `source` = `'shopify'`
   - `spendUsd` = NULL (Shopify has no spend — it is a revenue source)
   - `directRevenue`, `directConversions` = aggregated values above
   - `impressions`, `clicks`, `ctr`, `cpm` = NULL (not applicable)
4. Upsert into campaign_metrics using `onConflictDoUpdate` with target `[tenantId, campaignId, date, source]`
5. Update rawApiPulls: `normalized = true`, `normalizedAt = now()`, `schemaVersion = '1.0'`
6. Return count of date-rows upserted

**Export `processShopifySync(params: { tenantId, integrationId, dateRange })`:**
1. Get connector config (decrypt tokens)
2. Determine sync path: if dateRange > 30 days, use `fetchMetricsBulk`; otherwise use `fetchMetrics`
3. Ensure synthetic "shopify-revenue" campaign exists for this tenant (upsert into campaigns table)
4. Store raw pull, normalize, update ingestion coverage
5. Return summary: { recordsIngested, datesProcessed }

Validate with Zod: order amounts are numeric strings, processedAt is valid ISO date.
Use `withTenant(tenantId, ...)` for all database operations.

**Currency handling:** For v1, store `totalPriceSet.shopMoney.amount` in the shop's default currency. Currency normalization (multi-currency stores) is a v2 concern. The `shopMoney` accessor always returns shop currency, not presentment currency, which is the correct choice.
