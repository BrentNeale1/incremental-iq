**packages/ingestion/src/scheduler/queues.ts:**
Define BullMQ queue infrastructure:
- Create Redis connection config from env vars (`REDIS_HOST`, `REDIS_PORT`, `REDIS_PASSWORD`)
- Export `ingestionQueue = new Queue('ingestion', { connection })` — single queue for all sync/backfill jobs
- Export `registerNightlySync(tenantId: string, platform: Platform)` — calls `queue.upsertJobScheduler()` per RESEARCH.md Pattern 2:
  ```typescript
  await ingestionQueue.upsertJobScheduler(
    `nightly-${platform}-${tenantId}`,
    { pattern: '0 2 * * *' },  // 2am UTC daily (user decision: daily overnight sync)
    {
      name: 'incremental-sync',
      data: { tenantId, platform, integrationId, type: 'incremental' },
      opts: { attempts: 3, backoff: { type: 'exponential', delay: 60_000 }, removeOnComplete: { count: 10 }, removeOnFail: { count: 50 } },
    }
  );
  ```
- Export `enqueueBackfill(tenantId, platform, integrationId, dateRange)` — adds a one-time backfill job
- Export `enqueueManualSync(tenantId, platform, integrationId)` — adds a one-time manual sync job
- `upsertJobScheduler` is idempotent — safe to call on every deploy or integration connect

**packages/ingestion/src/scheduler/jobs/sync.ts:**
Export `processSyncJob(job: Job<SyncJobData>)`:
1. Create a `syncRuns` record with status `'running'`
2. Look up integration from DB, decrypt credentials
3. Get the appropriate platform process function:
   - `meta` -> `processMetaSync`
   - `google_ads` -> `processGoogleAdsSync`
   - `shopify` -> `processShopifySync`
4. Calculate date range for incremental sync: yesterday (since last sync). If `lastSyncedAt` is null, treat as first sync and enqueue a backfill job instead.
5. Call the process function with the date range
6. On success: update syncRuns status to `'success'`, update integrations.lastSyncedAt and lastSyncStatus
7. On partial failure (user decision: keep successfully pulled data): update syncRuns status to `'partial'`, log error message, integrations.lastSyncStatus = 'partial'
8. On full failure: update syncRuns status to `'failed'`, log error message, integrations.lastSyncStatus = 'failed'. If token expired, set integration status to `'expired'`.
9. Each platform syncs independently (user decision) — a Meta failure does not affect Google Ads or Shopify jobs

**packages/ingestion/src/scheduler/jobs/backfill.ts:**
Export `processBackfillJob(job: Job<SyncJobData>)`:
1. Create syncRuns record with `runType: 'backfill'`, status `'running'`
2. Determine the backfill date range:
   - Auto-max: Go back as far as the API allows (user decision: auto-max backfill on first connection)
   - Meta: 37 months for aggregate totals (RESEARCH.md Pitfall 2)
   - Google Ads: up to 3 years (no practical limit)
   - Shopify: all available order history with `read_all_orders` scope
3. Chunk the date range by month using `date-fns` `eachMonthOfInterval` (per RESEARCH.md Pattern 3)
4. Process each month sequentially:
   ```typescript
   for (let i = 0; i < months.length; i++) {
     const monthStart = format(months[i], 'yyyy-MM-dd');
     const monthEnd = format(endOfMonth(months[i]), 'yyyy-MM-dd');
     await processFunction({ tenantId, integrationId, dateRange: { start: monthStart, end: monthEnd } });
     await job.updateProgress({ completed: i + 1, total: months.length, unit: 'months' });
     // UI reads: "Meta Ads: 14 of 36 months pulled" (user decision: live progress)
   }
   ```
5. On partial failure (month fails): log the failed month in progressMetadata, continue with next month (user decision: keep successfully pulled data, retry rest on next cycle)
6. Update syncRuns on completion with recordsIngested total and final status
7. After backfill completes, register the nightly scheduler (`registerNightlySync`) so future syncs run automatically
8. Check ingestion_coverage to determine if 1 year of data is now available — if so, update tenants.analysisUnlocked = true (ARCH-03 gate)

**packages/ingestion/src/scheduler/workers.ts:**
Worker process entry point — meant to be run as a separate Node.js process:
```typescript
import { Worker } from 'bullmq';
const worker = new Worker('ingestion', async (job) => {
  if (job.name === 'incremental-sync' || job.name === 'manual-sync') {
    return processSyncJob(job);
  } else if (job.name === 'backfill') {
    return processBackfillJob(job);
  }
}, { connection, concurrency: 3 });
```
- Concurrency 3: allows parallel sync of different tenants/platforms
- Log worker lifecycle events (ready, failed, completed)
- Handle graceful shutdown on SIGTERM/SIGINT

**apps/web/app/api/integrations/[id]/sync/route.ts — POST handler:**
Manual "Sync now" endpoint (user decision: available per integration, rate-limited):
1. Read integration ID from route params
2. Read tenantId from request (header or session — for Phase 2 use a header since auth is Phase 6)
3. Rate limit: Check sync_runs table for this integration — if a manual sync was triggered in the last hour, return 429 Too Many Requests with `Retry-After` header. Allow max 3 manual syncs per day per integration (Claude's discretion on exact limit).
4. Look up integration from DB to get platform
5. Call `enqueueManualSync(tenantId, platform, integrationId)` to add job to BullMQ queue
6. Return 202 Accepted with `{ message: 'Sync queued', jobId }`
7. Does NOT wait for sync to complete — the job runs asynchronously in the worker

**apps/web/app/api/integrations/[id]/status/route.ts — GET handler:**
Per-integration status and sync history (user decision: freshness badge + sync history log):
1. Read integration ID from route params, tenantId from header
2. Query integrations table for status, lastSyncedAt, lastSyncStatus
3. Query sync_runs table for the last 7 runs (user decision: 5-7 sync history entries) ordered by startedAt DESC
4. Calculate freshness: time since lastSyncedAt as human-readable string ("2h ago", "1d ago")
5. If a sync or backfill is currently running (status = 'running' in sync_runs): include `syncInProgress: true` and the progress metadata
6. Return JSON:

}
