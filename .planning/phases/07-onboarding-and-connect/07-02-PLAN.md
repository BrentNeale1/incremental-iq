    **1. Create onboarding route group layout:**
    Create `apps/web/app/(onboarding)/layout.tsx` — a minimal layout with NO dashboard sidebar/header.
    - Import TenantProvider from `@/lib/auth/tenant-context`
    - Import QueryProvider from `@/components/layout/QueryProvider`
    - Import auth from `@/auth`, headers from `next/headers`, redirect from `next/navigation`
    - Server component: call `auth.api.getSession({ headers: await headers() })`, redirect to `/login` if no session
    - Wrap children in `<TenantProvider tenantId={session.user.tenantId}><QueryProvider>...</QueryProvider></TenantProvider>`
    - Minimal layout: `<div className="min-h-screen bg-background flex flex-col">{children}</div>`
    - Import and apply `@/app/globals.css` (via import statement at top)

    **2. Create onboarding page (server component):**
    Create `apps/web/app/(onboarding)/onboarding/page.tsx`.
    - Import auth, headers, redirect, db, tenants, eq
    - Call `auth.api.getSession()` — redirect to `/login` if no session
    - Query tenants table: `db.select({ onboardingCompleted: tenants.onboardingCompleted }).from(tenants).where(eq(tenants.id, session.user.tenantId)).limit(1)`
    - If `rows[0]?.onboardingCompleted` is true → `redirect('/')` (returning onboarded user)
    - Render `<OnboardingWizard />`

    **3. Update middleware to allow /onboarding for authenticated users:**
    In `apps/web/middleware.ts`, the current logic redirects authenticated users on auth routes to `/`. The `/onboarding` route does NOT need special middleware treatment — authenticated users have a session cookie, so middleware passes them through. However, we need to ensure middleware does NOT redirect authenticated users FROM /onboarding to / (it won't, because /onboarding is not listed as an isAuthRoute). Verify no change is needed. If the current middleware logic already allows /onboarding for authenticated users, leave it unchanged.

    **4. Create WizardStepper component:**
    Create `apps/web/components/onboarding/WizardStepper.tsx` — a 'use client' component.

    - Display 4 numbered steps in a horizontal bar: "1. Connect", "2. Events", "3. Markets", "4. Mode"
    - Current step highlighted with primary color, completed steps with check mark, future steps dimmed
    - Use a flex row with separator lines between steps
    - Use shadcn-compatible styling (Tailwind v4 classes)

    **5. Create IntegrationCard component:**
    Create `apps/web/components/onboarding/IntegrationCard.tsx` — a 'use client' component.
    - Props: `platform: string, name: string, description: string, icon: React.ReactNode, category: 'commerce' | 'paid_channel', status: 'disconnected' | 'connecting' | 'connected' | 'error', onConnect: () => void, errorMessage?: string`
    - Use shadcn Card component
    - States:
      - `disconnected`: Show "Connect" button (primary variant)
      - `connecting`: Show spinner/loading state, button disabled
      - `connected`: Show green check + "Connected" badge, button hidden
      - `error`: Show red error message inline + "Retry" button
    - Shopify special case: when platform is 'shopify' and status is 'disconnected', show an Input field for shop domain (e.g., "mystore.myshopify.com") above the Connect button. The `onConnect` callback receives the shop domain as a parameter. Add an optional `onConnectWithShop?: (shop: string) => void` prop for this.
    - Card layout: icon top-left, name + description, status/button bottom

    **6. Create IntegrationConnectStep component:**
    Create `apps/web/components/onboarding/IntegrationConnectStep.tsx` — a 'use client' component.
    - Props: `connectedIntegrations: Record<string, { integrationId: string }>, onIntegrationConnected: (platform: string, integrationId: string) => void`
    - Import useTenantId from `@/lib/auth/tenant-context`
    - Categorize integrations into two groups per CONTEXT.md decision:
      - Commerce/Analytics sources: Shopify, GA4
      - Paid channels: Meta Ads, Google Ads
    - Render section headers for each category, then IntegrationCard for each platform
    - For each card, implement `handleConnect(platform)`:
      - CRITICAL: `window.open()` MUST be called synchronously in the click handler (Pitfall 8 — popup blocker). Open BEFORE any async work.
      - Build OAuth URL: `/api/oauth/${platform}?tenantId=${tenantId}` (OAuth routes still need tenantId query param — they were excluded from Phase 6 session retrofit)
      - For Shopify: URL is `/api/oauth/shopify?tenantId=${tenantId}&shop=${shopDomain}`
      - Open popup: `window.open(url, 'oauth', 'width=600,height=700,scrollbars=yes')`
      - Set card status to 'connecting'
      - Listen for postMessage: `window.addEventListener('message', handler)`
      - Handler checks `event.origin === window.location.origin` and `event.data?.type === 'oauth_complete'` and `event.data?.platform === platform`
      - On success: update card to 'connected', call `onIntegrationConnected(platform, event.data.integrationId)`
      - Also detect popup closed without completing: poll `popup.closed` every 500ms via setInterval, clean up listener on close
      - On error or popup closed: set card status to 'error' with message
    - Display validation message at bottom: "Connect at least one commerce/analytics source and one paid channel to continue"
    - Expose a computed `canProceed` value: at least one commerce/analytics AND at least one paid channel connected

    **7. Create OnboardingWizard component:**
    Create `apps/web/components/onboarding/OnboardingWizard.tsx` — a 'use client' component.
    - State interface:
      ```typescript
      interface WizardState {
        currentStep: 1 | 2 | 3 | 4;
        connectedIntegrations: Record<string, { integrationId: string }>;
        ga4IntegrationId: string | null;
        ga4PropertyId: string | null;
        ga4EventsSelected: boolean;
        marketsConfirmed: boolean;
        outcomeMode: 'ecommerce' | 'lead_gen' | null;
        isCompleting: boolean;
        showTransition: boolean;
      }
      ```
    - On mount: fetch `GET /api/onboarding/status` to restore state for resume-on-return (CONTEXT.md: "Mid-onboarding abandonment: state persisted, user resumes where they left off on return"). Map response to WizardState, set currentStep to suggestedStep.
    - Also fetch `GET /api/integrations/status` to get connected integrations list with their platform and ID.
    - Render WizardStepper at top with currentStep and completedSteps derived from state
    - Render current step content:
      - Step 1: `<IntegrationConnectStep>` — Next enabled when canProceed (one commerce + one paid)
      - Step 2: If GA4 connected, render GA4 property selector + `<GA4EventSelector>`. If GA4 NOT connected, show message "GA4 not connected — you can add it later in Settings" with a Skip button. Next enabled when GA4 events selected OR GA4 not connected (skip allowed).
        - GA4 property selector: fetch `GET /api/ga4/properties?integrationId=${ga4IntegrationId}`. Display dropdown. If only one property, auto-select. Pass selected propertyId to GA4EventSelector.
      - Step 3: Render `<MarketConfirmationStep>` — but wrap it so that save happens on Next click, not per-action. Implementation: the existing MarketConfirmationStep already saves per-action via fetch. Per CONTEXT.md "Batch save — all edits saved when user clicks Next". Override approach: since the existing component does per-action saves to the API directly, and the API is idempotent, allow the component to function as-is for v1. The "batch save" intent is functionally met because all edits are persisted before the user moves to the next step. Next enabled when markets exist (at least 1 market row present).
      - Step 4: Render `<OutcomeModeSelector>` — pass `onSelect` callback that updates wizard state. Next becomes "Complete" button.
    - Back and Next buttons at bottom of each step:
      - Back: navigate to previous step (disabled on Step 1)
      - Next: validate step completion, advance to next step
      - On Step 4 "Complete": POST to `/api/onboarding/complete`, set `showTransition: true` → show OnboardingTransition
    - All step state managed in wizard — step components report completion via callbacks
    - Show loading skeleton while initial status fetch is in progress
