    **GET handler:**
    1. Accept `?tenantId=` query param
    2. Query markets table for tenant, ordered by campaignCount DESC
    3. Return JSON array: `{ id, countryCode, displayName, campaignCount, isConfirmed, createdAt }`
    4. Include campaign counts as confidence indicators per user decision ("AU -- 87 campaigns", "US -- 243 campaigns")

    **PUT handler:**
    1. Accept JSON body: `{ markets: Array<{ id?: string, countryCode: string, displayName: string, action: 'confirm' | 'rename' | 'merge' | 'add' | 'delete' }> }`

    4. For 'merge': reassign all campaign_markets from source marketId to target marketId, delete source market row

    6. For 'delete': reassign campaigns to Global/Unassigned (marketId=NULL), delete market row
    7. All operations via withTenant() for RLS
    8. Return updated markets list

    2. Import and call `detectMarketsForTenant(tenantId)` from `@incremental-iq/ingestion`
    3. This triggers the full market detection pipeline (Google Ads geo + Meta ad set geo) from Plan 01
    4. Return the detected markets array
    5. This endpoint is called during onboarding after ad accounts are connected, and can be called again to re-detect

    **Extend `packages/ingestion/src/scoring/worker.ts` — processScoringJob():**

    The current worker fetches ALL campaign metrics and sends them to Python without market segmentation. This causes cross-market contamination (RESEARCH.md Pitfall 6).

    Changes:
    1. After loading campaign metrics for a campaign, query `campaign_markets` to get the campaign's marketId
    2. If marketId is not NULL: filter campaign_metrics by market before sending to Python
       - The filtering approach: for spend-source campaigns (Meta, Google Ads), their metrics are already campaign-specific so no filtering needed
       - For outcome-source campaigns (Shopify, GA4), the revenue/leads are currently whole-tenant — when marketId is available, partition by multiplying by the market's campaign spend share (spend_market / spend_total) as a proxy
       - CRITICAL: Do NOT add market filtering inside the Python sidecar. Filter metrics BEFORE sending to Python (RESEARCH.md Anti-Patterns)
    3. Pass marketId to persistScores() so it populates the STAT-05 scaffold column
    4. For tenants with multiple markets: run separate scoring calls per market (one Python sidecar call per market per campaign)
    5. Also keep a combined-market score (marketId=NULL) for overall rollups and single-market tenants

    Also check tenant.outcomeMode when building MetricRow arrays:
    - If outcomeMode='lead_gen': use directConversions as the revenue field in MetricRow (RESEARCH.md Pattern 10)
    - If outcomeMode='ecommerce': use directRevenue as before

    **Extend `packages/ingestion/src/scoring/persist.ts` — persistScores():**
    1. Accept optional `marketId: string | null` parameter
    2. Pass marketId to incrementalityScores insert (currently always NULL from Phase 3 scaffold)
    3. This fills in the STAT-05 scaffold

    **Extend `packages/ingestion/src/scoring/rollup.ts` — computeHierarchyRollups():**
    The Phase 3 rollup already has STAT-05 scaffold that appends marketId to group keys when non-null.
    1. Verify the existing scaffold works: when marketId is populated on score rows, rollups should naturally group by market
    2. Add a market-level rollup (all campaigns in same market) as a new level between channel and overall
    3. Overall rollup (level 4) should aggregate across all markets — this is the "All Markets" view

    **Extend `packages/ingestion/src/scheduler/jobs/sync.ts`:**
    After a successful sync that triggers scoring, also trigger market detection re-run if new campaigns were synced. This ensures new campaigns get market assignments before scoring.
    - Check if any new campaign rows were created during this sync (compare campaign count before/after)
    - If yes, call detectMarketsForTenant() before enqueueing scoring jobs
