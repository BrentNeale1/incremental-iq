Purpose: Eliminate the IDOR vulnerability where any user can pass any tenantId as a query parameter. After this plan, tenant data isolation is enforced by the auth session — users can only access their own tenant's data.
Output: All API routes secured with session-based auth, all dashboard pages wired to session identity.
</objective>

  <action>
Apply Pattern 8 from RESEARCH.md to ALL existing API routes. For each route file:

1. Add imports at the top:
   ```typescript
   import { auth } from "@/auth";
   import { headers } from "next/headers";
   ```

2. At the start of each handler function (GET, POST, PUT, PATCH, DELETE), add session validation:
   ```typescript
   const session = await auth.api.getSession({ headers: await headers() });
   if (!session) {
     return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
   }
   const tenantId = session.user.tenantId;
   ```

3. Remove `tenantId` from searchParams extraction. Remove the `!tenantId` check from the required params validation (it now comes from session, not the request).

4. Keep all other query parameters (from, to, compareFrom, compareTo, marketId, level, platform, etc.) — only tenantId changes source.

5. IMPORTANT: Do NOT modify the OAuth callback routes (`apps/web/app/api/oauth/*/callback/route.ts` and `apps/web/app/api/oauth/*/route.ts`) — these are used during the OAuth flow when the user may not yet be authenticated, and they use their own tenant resolution mechanism.

Apply this pattern consistently to ALL 16 non-OAuth API routes listed in files above. The pattern is identical for every route — only the import and the first 4 lines of each handler change.

SECURITY NOTE from RESEARCH.md: Always use `session.user.tenantId` for data access. Never trust a client-provided tenantId. This eliminates the IDOR vulnerability that exists today.
