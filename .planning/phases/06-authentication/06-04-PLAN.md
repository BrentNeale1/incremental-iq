  <action>
    Fix three distinct bugs identified in UAT testing:

    **1. Sign-up redirect swallowed (signup/page.tsx):**
    The `catch {}` block on line 87 swallows the NEXT_REDIRECT error that Next.js `redirect()` throws from the server action. Import `isRedirectError` from `next/dist/client/components/redirect-error` (the canonical internal import for Next.js 15 App Router). In the catch block, check `if (isRedirectError(e)) { throw e; }` before setting the generic error message. This allows the redirect to propagate while still catching genuine errors.

    The catch block currently looks like:
    ```
    } catch {
      setError('Something went wrong. Please try again.');
    }
    ```

    Change it to:
    ```
    } catch (e) {
      if (isRedirectError(e)) throw e;
      setError('Something went wrong. Please try again.');
    }
    ```

    Add the import at the top:
    ```
    import { isRedirectError } from 'next/dist/client/components/redirect-error';
    ```

    **2. Login missing callbackURL (login/page.tsx):**
    The `authClient.signIn.email()` call on line 41 does not include a `callbackURL` parameter. Without it, Better Auth does not know where to redirect after setting the session cookie, and the client-side `router.push('/')` fires before the cookie is fully set, resulting in a redirect loop back to /login. Add `callbackURL: '/'` to the signIn.email options object:

    Change:
    ```
    const { error: signInError } = await authClient.signIn.email({
      email,
      password,
    });
    ```

    To:
    ```
    const { error: signInError } = await authClient.signIn.email({
      email,
      password,
      callbackURL: '/',
    });
    ```

    **3. Forgot-password autofill bypass (forgot-password/page.tsx):**
    The submit button uses `disabled={isPending || !email}` (line 98). When the browser autofills the email field, it sets the DOM input value visually but does NOT fire React's synthetic `onChange` event, so the `email` state stays as empty string `''`. The button remains disabled even though the user sees an email in the field.

    Add an `onBlur` handler to the email Input that syncs the current DOM value into React state. This catches the autofill scenario because the blur event fires after autofill populates the field.

    Change the email Input (around line 91) from:
    ```
    <Input
      id="email"
      type="email"
      placeholder="you@company.com"
      value={email}
      onChange={(e) => setEmail(e.target.value)}
      ...
    />
    ```

    To:
    ```
    <Input
      id="email"
      type="email"
      placeholder="you@company.com"
      value={email}
      onChange={(e) => setEmail(e.target.value)}
      onBlur={(e) => setEmail(e.target.value)}
      ...
    />
    ```

    This ensures that if the browser autofills the email field, clicking anywhere (or tabbing to the submit button) triggers the blur and syncs the value into React state, enabling the submit button.

    Fix two infrastructure issues:

    **1. CSS resolution failure on cold reload (globals.css):**
    The `@import 'shadcn/tailwind.css'` on line 3 fails to resolve during dev server cold reloads triggered by middleware 302 redirects. The full page navigation forces a fresh CSS pipeline compile, and the `shadcn` package import resolution is unreliable in this scenario.

    Replace the `shadcn/tailwind.css` import with the direct CSS content that `shadcn/tailwind.css` provides. The shadcn Tailwind v4 package import essentially provides the Tailwind CSS theme configuration and base layer. Since the globals.css already contains all the CSS variables, oklch color definitions, and @theme inline block that shadcn needs, the `shadcn/tailwind.css` import is redundant — it was auto-added by `npx shadcn init` but its content is already inlined.

    Change line 3 from:
    ```
    @import "shadcn/tailwind.css";
    ```

    To remove it entirely. The file should start with:
    ```
    @import "tailwindcss";
    @import "tw-animate-css";
    ```

    The `@theme inline` block already contains all the shadcn CSS variable mappings (--color-background, --color-foreground, etc.) and the `:root` / `.dark` blocks define all values. This is the complete shadcn theme — the package import was a convenience wrapper around these same declarations.

    **2. Create .env.local with required auth environment variables:**
    Better Auth requires BETTER_AUTH_SECRET, BETTER_AUTH_URL, and NEXT_PUBLIC_APP_URL to function. Without these, auth.ts cannot sign session tokens and auth-client.ts cannot resolve API endpoints.

    Create `apps/web/.env.local` with:
    ```
    # Better Auth — required for session signing
    # Generate a new secret: openssl rand -base64 32
    BETTER_AUTH_SECRET=<generate a cryptographically random 32-byte base64 string using Node.js crypto.randomBytes(32).toString('base64')>
    BETTER_AUTH_URL=http://localhost:3000
    NEXT_PUBLIC_APP_URL=http://localhost:3000
    ```

    Generate the secret programmatically — do NOT use a hardcoded value. Use Node.js:
    ```
    node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
    ```

    NOTE: .env.local is gitignored (`.env.*` pattern in .gitignore), so this file will NOT be committed. This is correct — secrets should not be in version control. The file exists only on the user's local machine.
