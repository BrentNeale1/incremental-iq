<decisions_made>

- TimescaleDB pinned to 2.16.1 — version 2.17+ columnstore is fundamentally incompatible with RLS (cannot enable/disable RLS on compressed hypertables)
- Compression deferred in migration 0001 — TimescaleDB compression conflicts with RLS in all versions; production should disable RLS on hypertables and rely on application-layer tenant isolation
- raw_api_pulls primary key changed to composite (id, pulled_at) — TimescaleDB requires partition column in all unique indexes
- tenantId additionalField changed to input:true — input:false blocks the field even in server-side auth.api.signUpEmail(); server action still controls the value
- advanced.database.generateId set to false — lets PostgreSQL gen_random_uuid() generate IDs instead of Better Auth's nanoid (schema uses uuid columns)
</decisions_made>

<context>
The root cause of all 3 auth failures (login, signup, page routing) was the absence of a database.
Once the database was stood up, two additional config bugs surfaced:
1. Better Auth's `input: false` on tenantId prevented the server action from setting it during sign-up
2. Better Auth's default nanoid ID generation produced strings incompatible with the uuid column type
Both were fixed in auth.ts. The e2e test script confirmed all flows work at the API level.
Next step is browser-level UAT to confirm the full UI flow works end-to-end.
</context>
