  9. Update campaign_metrics modeled_* columns for the latest data points (ARCH-02: dual attribution in same row)
  10. Log job progress via job.updateProgress()

**packages/ingestion/src/scoring/rollup.ts:**

Types:
```typescript
interface CampaignScore {
  campaignId: string;
  funnelStage: 'awareness' | 'consideration' | 'conversion';
  platform: string; // 'meta' | 'google_ads' | 'shopify'
  spendUsd: number;
  liftMean: number;
  liftLower: number;
  liftUpper: number;
  confidence: number;
  status: string;
  marketId: string | null; // STAT-05 scaffold
}

interface RollupScore {
  level: 'cluster' | 'channel' | 'overall';
  groupKey: string; // e.g., 'meta_awareness', 'meta', 'overall'
  liftMean: number;
  liftLower: number;
  liftUpper: number;
  confidence: number; // confidence-weighted best estimate
  campaignCount: number;
  totalSpend: number;
}
```

Functions:

`spendWeightedRollup(scores: CampaignScore[]): RollupScore`
- Per RESEARCH.md: spend-weighted average for lift_mean
- Uncertainty propagation: variance-weighted combination where individual campaign variance = ((liftUpper - liftLower) / 4)^2
- Propagated variance = sum of (individual_variance * (spend_weight)^2)
- Rollup half-width = 2 * sqrt(propagated_variance) for ~95% interval
- Per user decision: "Uncertainty propagation: cluster/channel confidence intervals widen to reflect low-confidence campaigns"
- Also compute confidence-weighted "best estimate" per user decision: weight each campaign's lift by both spend AND confidence. Formula: weighted_lift = sum(lift_i * spend_i * confidence_i) / sum(spend_i * confidence_i). This gives marketers a directional signal even when some campaigns have low confidence.

`computeHierarchyRollups(scores: CampaignScore[]): RollupScore[]`
- Group campaigns by cluster key: `${platform}_${funnelStage}` (e.g., 'meta_awareness', 'google_ads_conversion')
- For each cluster group, call spendWeightedRollup → produces cluster-level scores
- Group clusters by channel key: `${platform}` (e.g., 'meta', 'google_ads')
- For each channel group, rollup the cluster scores → produces channel-level scores
- Compute overall rollup from all channel scores
- Return all rollup scores (clusters + channels + overall)
- Filter out campaigns with status != 'scored' before rollup (don't include insufficient_data campaigns in averages)

`recomputeRollups(db, tenantId: string): Promise<void>`
- Query all latest campaign scores for the tenant from incrementality_scores (most recent scoredAt per campaign)
- Query campaign details (funnelStage, platform) from campaigns table
- Build CampaignScore array, including total spend from campaign_metrics last 30 days
- Call computeHierarchyRollups()
- Persist rollup scores to incrementality_scores with campaignId = a well-known sentinel (e.g., the groupKey as campaignId, or a separate rollup_scores table approach — use campaignId convention: 'rollup:{level}:{groupKey}' to distinguish from campaign-level scores)
- Per STAT-05 scaffold: include marketId in grouping if non-null. For now, marketId is always null (Phase 5 populates it). When Phase 5 adds market data, the rollup logic naturally segments by market.

Update packages/ingestion/src/scoring/index.ts to export rollup functions.
