**A. Create SheetJS export helpers (apps/web/lib/export/excel.ts):**
- `exportToExcel(data: Record<string, unknown>[], filename: string)`: Creates XLSX workbook using XLSX.utils.json_to_sheet, writes as array buffer, triggers download via file-saver saveAs()
- `exportToCsv(data: Record<string, unknown>[], filename: string)`: Same data to CSV format via XLSX.utils.sheet_to_csv, triggers download
- Both functions work entirely client-side (no server round-trip)
- Pattern from RESEARCH.md Pattern 7

**B. Create ExportButton (apps/web/components/dashboard/ExportButton.tsx):**
- 'use client' directive
- Uses shadcn DropdownMenu with two options: "Export as CSV" and "Export as Excel"
- Download icon (lucide-react Download) as trigger button
- Receives `data` and `filename` props from the page component
- Data comes from TanStack Query cache (already filtered by current date range — Pitfall 7)
- Placed in AppHeader (visible on all pages) — exports data currently visible on the active page

**C. Create NotificationBell (apps/web/components/notifications/NotificationBell.tsx):**
- 'use client' directive
- Bell icon (lucide-react Bell) with red unread count badge
- Badge shows number if unread > 0, hides if 0
- On click: opens NotificationPanel
- Unread count from TanStack Query polling `/api/notifications?unreadOnly=true&tenantId=X` with staleTime: 60_000 (60 seconds — Pitfall 8)
- Also updates useNotificationStore().unreadCount for cross-component access

**D. Create NotificationPanel (apps/web/components/notifications/NotificationPanel.tsx):**
- 'use client' directive
- Uses shadcn Sheet (slide-over from right) on mobile, Popover on desktop
- Lists recent notifications (limit 50) ordered by createdAt DESC
- Each notification: type icon (Bell for recommendation, AlertTriangle for anomaly, Calendar for seasonal, Activity for data health), message text, relative time, unread indicator (blue dot)
- "Mark all as read" button at top
- Click a notification: navigate to linkPath and mark as read
- Empty state: "No notifications yet — we'll alert you when something needs attention"

**E. Create NotificationSettings (apps/web/components/notifications/NotificationSettings.tsx):**
- 'use client' directive
- Accessible from NotificationPanel footer or Data Health page
- Per notification type (anomaly, recommendation, seasonal, data_health):
  - Toggle: In-app notifications (on/off)
  - Toggle: Email notifications (on/off)
- Reads/writes from `/api/notifications/preferences` endpoint
- Per user decision: "Simple toggles in settings — per notification type, per channel (in-app vs email)"

**F. Update AppHeader to include NotificationBell and ExportButton:**
- Add NotificationBell to the right side of the header (before ThemeToggle)
- Add ExportButton to header — receives current page's exportable data via a context or prop callback
- Export data source: each page provides its own export data through a React context or export callback pattern

**G. Create email templates:**

`apps/web/emails/DataHealthAlert.tsx`:
- Uses @react-email/components (Html, Head, Body, Container, Text, Button, etc.)
- Props: integrationName, staleDays, reconnectUrl
- Subject: "Action required: {integrationName} data is {staleDays} days stale"
- Body: Brief message + prominent "Reconnect" button linking to reconnectUrl
- Per user decision: "Email notifications for data health issues"

`apps/web/emails/SeasonalDeadline.tsx`:
- Props: eventName, weeksUntil, recommendations (campaign names + actions)
- Subject: "{eventName} in {weeksUntil} weeks — prepare your campaigns"
- Body: Event name, timeline, per-campaign budget recommendation summary
- Per user decision: "Email notifications for seasonal deadlines only"

**H. Create notification generation backend (packages/ingestion/src/notifications/):**

`generate.ts`:
- `generateNotification(tenantId, type, message, linkPath)`: Insert into notifications table
- `checkAndNotifyDataHealth(tenantId)`: Query sync_runs for each integration, if last successful sync > 24h, generate 'data_health' notification + send email via email.ts
- `checkAndNotifySeasonalDeadlines(tenantId)`: Query seasonal_events for events within 6 weeks, if not already notified (check by event + tenant), generate 'seasonal' notification + send email
- `checkAndNotifyNewRecommendations(tenantId)`: After scoring run, compare new recommendations with previous; if new scale_up recommendations exist, generate 'recommendation' notification

`email.ts`:
- `sendDataHealthEmail(tenantEmail, integrationName, staleDays, reconnectUrl)`: Uses Resend API to send DataHealthAlert email template
- `sendSeasonalDeadlineEmail(tenantEmail, eventName, weeksUntil, recommendations)`: Uses Resend API to send SeasonalDeadline email template
- Reads RESEND_API_KEY from env, gracefully no-ops if not configured
- From address: `alerts@incremental-iq.com` (configurable via RESEND_FROM_EMAIL env var)

`index.ts`:
- Re-exports all notification functions

NOTE: These functions are called from the BullMQ scoring worker (packages/ingestion/src/scheduler/workers.ts) after scoring completes. The wiring into the worker is a lightweight addition: call `checkAndNotifyNewRecommendations` after `recomputeRollups` in the scoring worker's onComplete handler. Also call `checkAndNotifyDataHealth` from the ingestion worker's error handler.
