**A. Create types (apps/web/lib/recommendations/types.ts):**

```typescript
type RecommendationAction = 'scale_up' | 'watch' | 'investigate';
type RecommendationConfidenceLevel = 'high' | 'medium' | 'low' | 'insufficient';

interface Recommendation {
  id: string;
  campaignId: string;
  campaignName: string;
  platform: string;
  action: RecommendationAction;
  confidenceLevel: RecommendationConfidenceLevel;
  // Scale-up specifics (when action === 'scale_up')
  budgetIncreasePct?: number;
  currentDailySpend?: number;
  proposedDailySpend?: number;
  durationWeeks?: number;
  expectedIncrementalRevenue?: number;
  // Statistical detail (analyst view)
  liftMean?: number;
  liftLower?: number;
  liftUpper?: number;
  confidence?: number;
  saturationPct?: number;
  // Low-confidence path
  nextAnalysisDate?: string; // ISO date string
  holdoutTestDesign?: HoldoutTestDesign;
  // Ranking
  expectedImpact: number;
  // Seasonal context
  seasonalAlert?: SeasonalAlert;
}

interface HoldoutTestDesign {
  holdbackPct: number;
  durationWeeks: number;
  estimatedSampleSize: number;
  description: string; // human-readable: "Hold back 10% of spend for 2 weeks"
}

interface SeasonalAlert {
  eventName: string;
  weeksUntil: number;
  message: string; // "BFCM in 6 weeks: Campaign X scaled +40% last year"
  historicalLiftPct?: number;
}
```

**B. Create recommendation engine (apps/web/lib/recommendations/engine.ts):**

Key constants:
- `SCALE_UP_CONFIDENCE_THRESHOLD = 0.65` (65% minimum for scale-up)
- `SATURATION_HEADROOM_CUTOFF = 0.80` (80% saturation = no more headroom)
- `MAX_BUDGET_INCREASE_PCT = 50` (cap at 50% increase)
- `DEFAULT_DURATION_WEEKS = 3`

Functions to implement:

1. `computeBudgetRecommendation(currentSpendDaily, hillAlpha, hillMu, hillGamma, saturationPct, liftMean)`:
   - Returns `{ budgetIncreasePct, durationWeeks, expectedRevenue, proposedDailySpend }` or null
   - If saturationPct >= 0.80, return null (no headroom)
   - Scale to 75th percentile of remaining headroom, capped at 50% increase
   - Revenue via Hill function: `alpha * x^gamma / (mu^gamma + x^gamma)`
   - Expected incremental revenue = `(proposedRevenue - currentRevenue) * liftMean * 7 * durationWeeks`
   - Pattern from RESEARCH.md computeBudgetRecommendation

2. `classifyRecommendation(score, saturation)`:
   - If score.status === 'insufficient_data': action='investigate', confidenceLevel='insufficient'
   - If confidence < SCALE_UP_CONFIDENCE_THRESHOLD: action='watch', confidenceLevel='low', compute nextAnalysisDate (7 days from now), holdoutTestDesign
   - If confidence >= threshold AND saturation available: action='scale_up', compute budget recommendation
   - If confidence >= threshold AND saturation NULL: action='scale_up' but use lift-based estimate only (no specific budget numbers) per Pitfall 4

3. `generateRecommendations(tenantId)`:
   - Query latest adjusted incrementality_scores per campaign (DISTINCT ON scored_at DESC, WHERE score_type='adjusted')
   - CRITICAL: Filter OUT rollup sentinel rows (WHERE campaignId NOT LIKE 'rollup:%') per Pitfall 3
   - LEFT JOIN saturation_estimates for Hill curve parameters
   - LEFT JOIN campaigns for campaign name and platform
   - Compute 30-day average daily spend from campaign_metrics
   - For each campaign: classify recommendation, compute budget if applicable
   - Merge seasonal alerts from seasonal.ts
   - Sort by expectedImpact DESC (highest incremental revenue first per user decision)
   - Return Recommendation[]

CRITICAL per user decision (RECC-06): holdoutTestDesign is ONLY populated when confidence < threshold. The UI must check `recommendation.holdoutTestDesign` existence to decide whether to show the holdout option. The engine guarantees this field is absent on high-confidence recommendations.

Use `withTenant(tenantId)` pattern from existing codebase for all DB queries (RLS requires tenant context).

- Look up historical performance: query incrementality_scores for the same calendar period last year (if exists)
- Return SeasonalAlert[] sorted by date ascending

**B. GET /api/recommendations/route.ts:**

- Call `generateRecommendations(tenantId)`
- Return JSON array of Recommendation objects
- 200 OK with recommendations, 400 if missing tenantId

**C. GET /api/dashboard/kpis/route.ts:**
- Parse query params: `tenantId`, `from` (ISO date), `to` (ISO date), `compareFrom` (optional), `compareTo` (optional)
- Aggregate from campaign_metrics for date range: SUM(spendUsd), SUM(directRevenue), computed ROAS (revenue/spend), SUM(incrementalRevenue from modeled columns)
- Compute lift % as (SUM modeled incremental / SUM direct revenue) * 100
- If comparison range provided, compute same aggregates for comparison period and include deltas (current - comparison, percentage change)
- Return: `{ spend, revenue, roas, incrementalRevenue, liftPct, comparison?: { spendDelta, revenueDelta, roasDelta, ... } }`

**D. GET /api/dashboard/campaigns/route.ts:**
- Parse query params: `tenantId`, `from`, `to`, `platform` (optional filter), `level` (optional: 'campaign' | 'cluster' | 'channel' | 'overall')
- For level='campaign' (default): Query campaigns JOIN campaign_metrics (aggregated for date range) LEFT JOIN latest incrementality_scores (adjusted, score_type='adjusted')
- CRITICAL: Filter OUT rollup sentinel rows for campaign level
- For level='cluster'|'channel'|'overall': Query rollup rows FROM incrementality_scores WHERE campaignId LIKE 'rollup:{level}:%'
- Return array of campaign/rollup objects with: id, name, platform, funnelStage, spend, revenue, roas, liftMean, liftLower, liftUpper, confidence, status

**E. GET /api/dashboard/incrementality/route.ts:**
- Parse query params: `tenantId`, `campaignId` (optional — specific campaign detail), `scoreType` ('adjusted' | 'raw', default 'adjusted')
- If campaignId: return time series of scores for that campaign (all scored_at dates)
- If no campaignId: return latest score per campaign (for overview)
- Include saturation data joined from saturation_estimates
- Return typed IncrementalityDetail[]

**F. GET /api/dashboard/seasonality/route.ts:**
- Parse query params: `tenantId`, `months` (default 6 — how far forward to look)
- Query seasonal_events for upcoming events (system + tenant-specific)
- For each event, query historical campaign_metrics for the event's date range in prior years
- Return: `{ upcoming: SeasonalEvent[], historical: HistoricalPerformance[] }`

**G. GET /api/dashboard/saturation/route.ts:**
- Parse query params: `tenantId`, `campaignId` (optional)
- Query saturation_estimates joined with campaigns
- Return Hill curve parameters + current saturation position for chart rendering
- If campaignId specified, return detailed curve data points for chart

**H. GET + PATCH /api/notifications/route.ts:**
- GET: Parse tenantId, optional `unreadOnly=true` — query notifications table ordered by createdAt DESC, limit 50
- PATCH: Accept `{ ids: string[], read: true }` — mark notifications as read
- Return notification objects with id, type, message, linkPath, read, createdAt

**I. GET + PUT /api/notifications/preferences/route.ts:**
- GET: Return notification preferences from user_preferences table for tenantId
- PUT: Accept updated notification prefs, upsert into user_preferences table
- Return updated prefs

All routes use `withTenant()` for RLS. All routes validate required query params and return 400 for missing/invalid input.
