**Part A — Forecast API route** (CREATE `apps/web/app/api/dashboard/forecast/route.ts`):

1. Create a GET handler with session auth (same pattern as other dashboard API routes: `auth.api.getSession({ headers: await headers() })`, extract `tenantId` from session).
2. Accept `campaignId` query parameter (required — return 400 if missing).
3. Fetch campaign metrics from DB for the last 365 days:
   - Use `withTenant(tenantId, async (tx) => ...)` pattern.
   - Select `date`, `spendUsd`, `directRevenue`, `directConversions` from `campaignMetrics` table.
   - Filter by `tenantId`, `campaignId`, and `date >= cutoff` (365 days ago).
   - Order by `date`.
4. If fewer than 30 data points, return `{ historical: [], future: [] }` — insufficient data for meaningful forecast.
5. POST to Python FastAPI service at `${ANALYSIS_SERVICE_URL}/forecast`:
   - `ANALYSIS_SERVICE_URL` from `process.env.ANALYSIS_SERVICE_URL ?? 'http://localhost:8000'`
   - Body: `{ tenant_id, campaign_id, metrics: [...], forecast_days: 90 }`
   - Each metric: `{ date, spend_usd: parseFloat(m.spendUsd ?? '0'), revenue: parseFloat(m.directRevenue ?? '0'), conversions: parseFloat(m.directConversions ?? '0') }`
6. Parse Python response. The forecast response contains `forecast` array with `{ date, yhat, yhat_lower, yhat_upper }` points (verified in `packages/analysis/schemas/responses.py`).
7. Split into historical (date <= today) and future (date > today) segments.
8. Also return the actual campaign metrics as `actuals` array: `{ date, value: parseFloat(m.directRevenue ?? '0') }` for the historical date range — the chart needs BOTH the forecast fitted line AND the actual observed values.
9. Wrap the entire Python fetch in try/catch. On any network error or non-200 response, return `{ historical: [], future: [], actuals: [] }` — graceful degradation per CONTEXT locked decision and Pitfall 3.
10. Return `NextResponse.json({ historical, future, actuals })`.
