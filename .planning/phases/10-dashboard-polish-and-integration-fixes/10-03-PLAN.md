The root cause is a response shape mismatch. The API route (`/api/dashboard/saturation`) returns TWO shapes:
- Overview (no campaignId): `SaturationRow[]` — an array
- Detail (with campaignId): `{ campaign: SaturationRow, curvePoints: CurveDataPoint[], currentSpendLevel: number }` — an object

The hook always casts the response as `SaturationCurve[]`. When `campaignId` is provided, the response is an object, not an array, so `.find()` on the consuming page crashes.

**Fix in `apps/web/lib/hooks/useSaturation.ts`:**

1. Add a `SaturationDetailResponse` interface matching the API detail shape:
   ```typescript
   interface SaturationDetailResponse {
     campaign: {
       campaignId: string;
       campaignName: string | null;
       platform: string | null;
       saturationPct: number | null;
       hillAlpha: number | null;
       hillMu: number | null;
       hillGamma: number | null;
       status: string;
       estimatedAt: string;
     };
     curvePoints: Array<{ spendLevel: number; revenue: number; isCurrentSpend: boolean }>;
     currentSpendLevel: number;
   }
   ```

2. In the `queryFn`, after `res.json()`, normalize both shapes into `SaturationCurve[]`:
   ```typescript
   queryFn: async () => {
     const params = new URLSearchParams({
       ...(campaignId ? { campaignId } : {}),
     });
     const res = await fetch(`/api/dashboard/saturation?${params.toString()}`);
     if (!res.ok) {
       throw new Error(`Failed to fetch saturation: ${res.status}`);
     }
     const json = await res.json();
     // Normalize: detail mode returns { campaign, curvePoints, currentSpendLevel }
     // Overview mode returns SaturationCurve[]
     if (Array.isArray(json)) {
       return json as SaturationCurve[];
     }
     // Detail mode — extract campaign object and map to SaturationCurve shape
     const c = json.campaign;
     if (c) {
       return [{
         campaignId: c.campaignId,
         campaignName: c.campaignName ?? '',
         platform: c.platform ?? '',
         alpha: c.hillAlpha,
         mu: c.hillMu,
         gamma: c.hillGamma,
         saturationPercent: c.saturationPct,
         status: c.status as SaturationCurve['status'],
         scoredAt: c.estimatedAt,
       }] as SaturationCurve[];
     }
     return [] as SaturationCurve[];
   },
   ```

   Key mapping: API `hillAlpha` -> hook `alpha`, `hillMu` -> `mu`, `hillGamma` -> `gamma`, `saturationPct` -> `saturationPercent`, `estimatedAt` -> `scoredAt`. These match the existing `SaturationCurve` interface field names.

3. Keep the return type as `useQuery<SaturationCurve[]>` — consumers always get an array.

**Fix in `apps/web/app/(dashboard)/insights/page.tsx`:**

4. Add a defensive `Array.isArray` guard on the `selectedSaturation` useMemo (line 74-77) as a belt-and-suspenders safety net:
   ```typescript
   const selectedSaturation = React.useMemo(() => {
     if (!saturationData || !selectedRow) return null;
     if (!Array.isArray(saturationData)) return null;
     return saturationData.find((s) => s.campaignId === selectedRow.id) ?? null;
   }, [saturationData, selectedRow]);
   ```

   This guard ensures the page never crashes even if the hook normalization is bypassed or the API shape changes unexpectedly. The primary fix is in the hook; this is defense-in-depth.

Do NOT change the API route — it correctly serves two modes for different consumers. The hook is the right place to normalize.
